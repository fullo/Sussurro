How the pipeline works: from keypress to pasted text

A tour of the six on-device stages between holding a hotkey and text landing in your cursor.

6 min read · Concept

Sussurro's whole job happens between two events: you press a hotkey, and a few hundred milliseconds after you release it, cleaned-up text appears wherever your cursor is. Nothing in between touches the network unless you've pointed the cleanup step at Ollama — and even then, Ollama is a local server on your own machine. Here's what actually happens at each stage.

  1. Global hotkey press/release. A background listener (tauri-plugin-global-shortcut) watches for your configured combo — Ctrl+Shift+Space by default (⌘⇧Space on Mac). Sussurro supports both push-to-talk (hold to record, release to finish) and toggle mode; the same handler decides whether a press means start, finish, or should be ignored, based on whether a recording is already running.
  2. Microphone capture. On press, a dedicated thread opens the input device with cpal at whatever sample rate and channel count it natively runs at, and streams raw samples into a shared buffer. On release, that buffer is downmixed to mono and resampled — with a linear interpolator good enough for speech — to the 16 kHz mono format every STT engine expects. A short leading/trailing silence trim (RMS-based, a lightweight VAD) shaves off dead air before it ever reaches the model, and clips under roughly 0.3 seconds or with no real energy are dropped entirely rather than sent to inference — this is what keeps Whisper from hallucinating words like "thank you" out of pure silence.
  3. Local speech-to-text. The trimmed 16 kHz samples go to whichever engine you've picked in Settings: whisper.cpp (GPU-accelerated on supported platforms, any of the bundled model sizes) or NVIDIA Parakeet TDT v3 (a single ONNX model, CPU-optimized, auto-detecting across roughly two dozen European languages). Both run entirely on-device — see whisper-vs-parakeet.html for how to choose between them. The model is loaded once on first use and kept warm in memory rather than reloaded per dictation.
  4. Optional AI cleanup. The raw transcript is handed to your local Ollama model with a prompt built from your Cleanup level (None/Light/Medium/High), your personal dictionary, any per-app tone rule, and an output-language override if you've set one. If Ollama is unreachable, the request times out, or the model returns something that looks unrelated to what you said, Sussurro keeps the raw transcript instead of blocking or failing — cleanup is additive, never a hard dependency. See ai-cleanup-levels.html and cleanup-hallucination-guard.html for how that fallback and guard work.
  5. Paste-injection. The result is written to the system clipboard, a synthetic Ctrl+V (Cmd+V on Mac) is sent to whatever app currently has focus, and — after giving the target app a moment to read the clipboard — your previous clipboard contents are restored. Paste-injection works in far more apps than simulating individual keystrokes, and because the swap is save-then-restore, you never lose what was already on your clipboard. Linux/Wayland has its own dedicated path (the RemoteDesktop portal, with a native-tool fallback ladder) — see linux-injection.html and clipboard-injection.html.
  6. Local history. Both the raw transcript and the cleaned text are appended as one JSON line to a local history file — never sent anywhere. That's what powers full-text search and export later. See where-data-lives.html for the exact file locations, and searchable-history.html for what you can do with it.

The same pipeline also powers two variants worth knowing about. With streaming typing enabled, Sussurro doesn't wait for you to release the hotkey at all: it periodically re-transcribes the in-progress recording and either types stable new words directly (Cleanup None) or cleans and types complete sentences as they finish, holding back only the last few words that Whisper might still revise. When you release the hotkey, the final high-quality pass fills in just the tail. Command mode reuses the identical capture-and-transcribe steps, but treats the transcript as an instruction rather than dictation: it reads your current text selection via copy, asks the LLM to apply the instruction to it, and pastes the result back over the selection.

WHY IT MATTERS

Every stage above — capture, transcription, cleanup, injection, history — runs as a process on your machine. The only network call in the entire pipeline is the loopback request to your own Ollama instance for cleanup, and that step is fully optional and fails safe. There's no account, no server round-trip, and no dependency on connectivity to get from voice to pasted text.

Two details make the pipeline feel instantaneous rather than mechanical. First, heavy work — the STT pass and the Ollama call — happens on a background thread the moment the hotkey handler returns, so the global shortcut listener itself never blocks. Second, the model is loaded lazily on first use and then kept resident, so only your very first dictation after launch pays the load cost; every dictation after that starts straight from silence-trim to transcription.
