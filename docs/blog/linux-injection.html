Text injection on Linux: X11 and Wayland

How Sussurro types your dictation into whatever app has focus on Linux — the RemoteDesktop portal, the native tool fallbacks, and what to do when a paste silently fails.

5 min read · Setup guide

Two very different sessions, one API

Sussurro's usual way of injecting text is clipboard-based: save the clipboard, write the transcript, synthesize Ctrl+V (or Cmd+V on macOS), then restore the clipboard. On Linux, though, how the text actually reaches the app depends on whether the desktop runs X11 or Wayland — and on Wayland the primary path skips the clipboard entirely, typing the transcript directly through the RemoteDesktop portal. The two protocols have opposite security models, which is the whole reason Linux needs special handling.

X11

X11 has no permission model for input synthesis, so the keystroke needs no daemon and no consent dialog. It does need a clipboard helper, though — Sussurro pastes through the clipboard, and on Linux that only persists if an external tool holds the selection: sudo apt install xclip (or xsel). Without it, Copy and paste silently do nothing.

Wayland

Wayland compositors deliberately block apps from injecting input into other windows — that's a feature, not a bug, and it's exactly why KWin and Mutter don't expose the older virtual-keyboard protocol to arbitrary clients. Sussurro's primary path on Wayland is therefore the XDG RemoteDesktop portal: it asks the compositor's permission once, then types through a sanctioned channel that needs no daemon and no uinput access. This is the only zero-setup option on stock KDE and GNOME.

The first time Sussurro tries to inject, the OS shows a consent dialog for keyboard access. Approve it once and Sussurro keeps a restore token so later runs stay silent — one portal session is opened lazily and reused for the app's whole lifetime, so you're not re-prompted on every paste.

KDE QUIRK

KDE currently forgets RemoteDesktop grants across a reboot (kde#480235), so Plasma Wayland users may see the consent dialog again after restarting the session even though nothing changed on Sussurro's side. Approving it again is enough — the grant holds for the rest of that session.

The fallback ladder

If the portal isn't available — no xdg-desktop-portal running, or the compositor doesn't implement RemoteDesktop — Sussurro falls back to native tools, in order:

  • wtype, which speaks the virtual-keyboard protocol and covers wlroots compositors (Sway, Hyprland, river) as well as KDE Plasma.
  • ydotool, which works on any compositor including GNOME because it goes through uinput instead of a Wayland protocol — but it needs its daemon running.
  • enigo as a last resort, which can silently no-op on pure Wayland but still helps with XWayland apps.

The clipboard itself needs a helper that keeps holding the selection after Sussurro sets it: wl-copy (Wayland, from wl-clipboard) or xclip/xsel (X11). A transient in-process clipboard set reports success but empties immediately — so without one of these, Copy and the paste-restore step silently fail.

If paste doesn't work

  1. Dictate once and watch for the OS consent dialog — on Wayland it appears the first time Sussurro requests keyboard access. If you dismissed it by accident, dictate again to re-trigger it.
  2. On KDE, if injection suddenly stops working after a reboot, that's the known grant-reset above — approve the dialog again.
  3. The most common Wayland fix: after a dictation, press Ctrl+V by hand. If the dictated text pastes, the clipboard is fine and only the automatic keystroke is missing — the portal granted access but didn't deliver keys on your compositor. Install the native tool so Sussurro's fallback takes over: sudo apt install wtype for KDE/wlroots, or sudo apt install ydotool + systemctl --user enable --now ydotool for GNOME (your user needs /dev/uinput access, typically the input group). Confirmed on Ubuntu + KDE Plasma.
  4. For clipboard-only symptoms (paste works but the clipboard looks wrong afterward), install wl-clipboard: sudo apt install wl-clipboard.
  5. Check you're actually where you think you are: echo $XDG_SESSION_TYPE tells you X11 vs Wayland, which changes which of the above applies.

All of this — the portal integration, the tool ladder, and the clipboard fallbacks — is tracked in issue #40, which is the place to report a compositor or distro combination that still doesn't inject cleanly.
