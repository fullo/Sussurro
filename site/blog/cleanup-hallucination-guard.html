How cleanup avoids hallucinations

Small local models sometimes answer a dictation instead of cleaning it — here's the word-overlap guard and bilingual prompt that catch it.

5 min read · Concept

Cleanup runs a small local model — often a 2-3B model like llama3.2:3b — over every dictation to strip filler words and fix grammar. Small models are fast enough to run on a laptop CPU, but they have a specific failure mode: instead of cleaning a short transcript, they sometimes answer it. A dictation like "Proviamo l'audio. Va." (Italian for "Let's try the audio. Go.") has come back from the model as "Va bene, stiamo per iniziare." — "Okay, we're about to start." That's a reply to the text, not a cleaned version of it.

Sussurro treats this as an expected risk of small local models, not an edge case to hope around. Two mechanisms in cleanup/prompt.rs and cleanup/ollama.rs exist specifically to catch it.

The hallucination guard

looks_hallucinated() is a deterministic, non-LLM check that runs after cleanup for the Light and Medium levels. Its logic is simple: those two levels only remove fillers and fix grammar, so almost every word in the output should already exist in the input. The function lowercases and tokenizes both transcript and cleaned text, then measures what fraction of the output's words were actually present in the input:

  • Light requires at least 50% word overlap.
  • Medium, which tolerates lighter rewording, requires at least 30%.
  • An empty or punctuation-only output (like "…") is flagged outright.
  • High is never guarded, because it's designed to rewrite for brevity — every word can legitimately change.

When the guard trips, cleanup() in ollama.rs discards the model's output and keeps the raw transcript instead, logging cleanup output unrelated to transcript — keeping raw. The same fallback-to-raw path also covers Ollama being unreachable, a request timing out, or the model returning an empty string — cleanup is built to never fail the pipeline. Worst case, you get your unedited words back, not an error or someone else's sentence.

BY DESIGN

The guard is skipped whenever translation is requested, since translating legitimately replaces every word — comparing translated output against the original-language input would flag every honest translation as a hallucination.

Why the few-shot examples are bilingual

To stop small models from answering short, conversational-sounding dictations, build_messages() in prompt.rs appends two worked examples to the prompt before the real transcript: one English pair ("so um, we should review the uh quarterly numbers, right." → "So, we should review the quarterly numbers, right.") and one Italian pair, in that order, deliberately worded so they don't resemble typical dictations the model might pattern-match against.

Both languages are required, not one. An earlier version shipped only the Italian example, and it taught the 3B model to translate English dictations into Italian — the hallucination guard would then correctly catch that Italian output as unrelated to an English input and fall back to raw, silently disabling cleanup for every English dictation. Pairing an English example with the Italian one keeps the model anchored to the instruction that actually matters: clean in whatever language the speaker used. The examples are skipped entirely when a translation is requested, since same-language examples would fight the translation instruction.

The result is a cleanup step that fails safe twice over: a code-level guard that reverts to your actual words when the model drifts, and a prompt built from a bug report to stop that drift from being systematic.
